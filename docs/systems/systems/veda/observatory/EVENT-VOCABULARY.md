# VEDA Event Vocabulary

## Purpose

This document defines the current event vocabulary used by VEDA after Wave 2D.

It exists to give humans and LLMs a stable, readable explanation of what VEDA events mean, when they should be emitted, and how they relate to observatory state.

If this document and the Prisma schema ever disagree, the schema is authoritative and this document must be updated.

---

## Scope

This document covers the canonical observatory event vocabulary represented by:

- `EventLog`
- `EventType`
- `EntityType`
- `ActorType`

These event vocabularies are observatory-scoped.
They no longer describe the removed editorial and production workflows that existed before Wave 2D.

---

## Event Model

### `EventLog`

`EventLog` is VEDA's append-only audit ledger for event-eligible state changes.

Each event records:
- what happened (`eventType`)
- what kind of thing it happened to (`entityType`)
- which record it happened to (`entityId`)
- who or what performed the action (`actor`)
- which project it belongs to (`projectId`)
- when it happened (`timestamp`)
- optional structured details (`details`)

Event logs are historical truth.
They are not a scratchpad, not a cache, and not a general telemetry bucket.

## Actor Vocabulary

### `ActorType`

Current actor values are:

- `human`
- `llm`
- `system`

### Meaning

#### `human`
Use when the initiating action came from a human operator.

Examples:
- manually capturing a source
- updating an observatory setting
- creating a keyword target through an operator action

#### `llm`
Use when an LLM directly initiated the state change through an approved tool path.

Examples:
- an LLM-triggered workflow that creates a record through an allowed tool call
- an MCP-mediated update where the effective actor is the LLM

#### `system`
Use when the action was generated by internal system behavior rather than a direct human or LLM action.

Examples:
- an automated ingest or processing routine
- a background system mutation
- an internal maintenance transition

## Entity Vocabulary

### `EntityType`

Current VEDA entity types are:

- `sourceItem`
- `sourceFeed`
- `searchPerformance`
- `keywordTarget`
- `serpSnapshot`
- `cgSurface`
- `cgSite`
- `cgPage`
- `cgContentArchetype`
- `cgTopic`
- `cgEntity`
- `cgInternalLink`
- `cgSchemaUsage`
- `cgPageTopic`
- `cgPageEntity`
- `vedaProject`

### Meaning

This field tells the system what type of observatory object the event refers to.

It does not recreate the old PsyMetric-era editorial model.
In particular, removed types like production entities, draft artifacts, distribution events, and metric snapshots must not be silently reintroduced here.

### Usage rule

`entityType` must accurately describe the table/domain of the affected record.

`entityId` must refer to a real record of that type within the correct project scope.

## Event Definitions

### `SOURCE_CAPTURED`

Meaning:
A `SourceItem` was created through a capture flow.

Typical entity type:
- `sourceItem`

When to emit:
- when a source item is successfully captured and persisted

Do not use for:
- ordinary reads
- source display
- non-persisted parsing attempts

### `SOURCE_TRIAGED`

Meaning:
A source item changed triage-relevant status inside the observatory workflow.

Typical entity type:
- `sourceItem`

When to emit:
- when a source item's observatory status is intentionally changed as part of triage/use/archive handling

Note:
This is an observatory status event, not an editorial publishing event.

### `SYSTEM_CONFIG_CHANGED`

Meaning:
A global system configuration record was changed.

Typical entity type:
- system-level config references may still be logged under a suitable observatory entity mapping or implementation-specific convention

When to emit:
- when a `SystemConfig` record is updated in a meaningful way

### `PROJECT_CREATED`

Meaning:
A project-scoping container was created in VEDA.

Typical entity type:
- `vedaProject`

When to emit:
- when a new VEDA project record is created

Note:
This is about observatory scoping, not rich planning/orchestration lifecycle truth.

### `KEYWORD_TARGET_CREATED`

Meaning:
A keyword target was created.

Typical entity type:
- `keywordTarget`

When to emit:
- when a new governed keyword target enters the observation ledger

### `SERP_SNAPSHOT_RECORDED`

Meaning:
A SERP observation was recorded.

Typical entity type:
- `serpSnapshot`

When to emit:
- when a snapshot is successfully stored as an observation record

Do not use for:
- ranking calculations performed in memory without persistence
- planning-only keyword ideation

### `CG_SURFACE_CREATED`

Meaning:
A content-graph surface record was created.

Typical entity type:
- `cgSurface`

### `CG_SITE_CREATED`

Meaning:
A content-graph site record was created.

Typical entity type:
- `cgSite`

### `CG_PAGE_CREATED`

Meaning:
A content-graph page record was created.

Typical entity type:
- `cgPage`

### `CG_ARCHETYPE_CREATED`

Meaning:
A content-graph archetype record was created.

Typical entity type:
- `cgContentArchetype`

### `CG_TOPIC_CREATED`

Meaning:
A content-graph topic record was created.

Typical entity type:
- `cgTopic`

### `CG_ENTITY_CREATED`

Meaning:
A content-graph entity record was created.

Typical entity type:
- `cgEntity`

Note:
This is not the removed legacy publishing entity system.
It refers only to the current content-graph entity model.

### `CG_INTERNAL_LINK_CREATED`

Meaning:
A content-graph internal link record was created.

Typical entity type:
- `cgInternalLink`

### `CG_SCHEMA_USAGE_CREATED`

Meaning:
A content-graph schema usage record was created.

Typical entity type:
- `cgSchemaUsage`

### `CG_PAGE_TOPIC_CREATED`

Meaning:
A page-topic junction was created in the content graph.

Typical entity type:
- `cgPageTopic`

### `CG_PAGE_ENTITY_CREATED`

Meaning:
A page-entity junction was created in the content graph.

Typical entity type:
- `cgPageEntity`

## Emission Rules

### 1. Events are for durable state changes

Emit events for meaningful persisted state transitions.
Do not emit canonical `EventLog` rows for ordinary reads or non-persistent internal behavior.

### 2. State and event must agree

If an event claims a state change happened, the corresponding state must actually exist in the same committed transaction where required.

### 3. Event vocabulary must remain observatory-scoped

Do not reintroduce removed production/editorial events such as:
- draft lifecycle events
- blueprint workflow events
- distribution publishing events
- old entity workflow events
- metric snapshot events from removed domains

If a future capability needs new event types, those events must still fit VEDA's observatory-only ownership.

### 4. Cross-project contamination is forbidden

`projectId`, `entityType`, and `entityId` must describe a real record within the correct project scope.
Events must never imply cross-project state changes.

### 5. Details are supporting context, not the source of truth

`details` may enrich an event with structured metadata.
But the canonical meaning of the event must remain understandable from:
- `eventType`
- `entityType`
- `entityId`
- `actor`
- `projectId`
- `timestamp`

---

## What Was Removed in Wave 2D

The following event surfaces or adjacent vocabularies are no longer part of active VEDA truth:

- legacy publishing/entity events
- draft artifact lifecycle events
- blueprint proposal/application events
- distribution publishing events
- metric snapshot lifecycle events from removed domains
- quotable block events

If an old doc or script still references those, it is historical and must not be treated as current system truth.
